# Slash Command Scope

**最后更新**: 2026-04-16

## 背景

`gogo-app` 未来如果支持聊天输入框中的 slash 命令，需要先明确一个核心问题：

- 这些 slash 命令到底来自哪里
- `gogo-app` 应当向用户暴露哪些命令
- 哪些命令不应该进入 `gogo-app`

目前候选来源有两类：

1. `Pi agent` 已内置的 slash 命令
2. knowledge-base 中定义的 `skills` 与 `schemas`

这不是一个纯技术问题，更是一个产品边界问题。  
如果边界不清晰，slash 命令会迅速从“提升效率的快捷入口”变成“额外的系统复杂度”。

## 目标

我们希望 slash 命令满足以下目标：

- 服务知识库工作流，而不是把 `gogo-app` 变成 `Pi` 的通用控制台
- 让用户更容易发现、调用和复用知识库中的结构化能力
- 保持命令集合足够小、可理解、可维护
- 避免把底层 Agent runtime 的复杂度直接暴露给最终用户

## 候选来源

### 1. 直接暴露 Pi 内置 slash 命令

这条路线的含义是：

- `gogo-app` 识别并展示 `Pi` 已支持的 slash 命令
- 用户在 `gogo-app` 中把聊天框当成 `Pi` CLI 的延伸
- 命令语义主要由 `Pi` 决定，`gogo-app` 只负责透传、提示和展示

可能覆盖的能力包括：

- 会话级命令
- 登录/模型/上下文相关命令
- Agent runtime 或工具控制命令
- 其他 `Pi` 自带的系统命令

### 2. 暴露 knowledge-base 中的 skills

这条路线的含义是：

- slash 命令不代表“控制 Pi”
- slash 命令代表“触发知识库内的一个已定义能力”
- 命令本质是 knowledge-base 的技能入口

例如：

- `/summarize-paper`
- `/review-experiment`
- `/write-wiki-note`
- `/extract-dataset-schema`

这些命令更接近“知识工作流模板”，而不是底层 runtime 操作。

### 3. 暴露 knowledge-base 中的 schemas

这条路线的含义是：

- slash 命令不是执行技能，而是“按某个结构化 schema 发起任务”
- schema 更像受约束的任务表单或结构化输入模版

例如：

- `/new-insight`
- `/paper-card`
- `/experiment-log`

它们的主要价值在于把输入结构标准化，而不是直接提供完整技能行为。

## 方案对比

### A. 支持 Pi 内置 slash 命令

优点：

- 能力多，理论上与 `Pi` 原生体验更一致
- 如果用户本来熟悉 `Pi`，迁移成本低
- 某些底层命令不需要 `gogo-app` 重做一遍 UI

缺点：

- 命令集合会明显变大，认知负担高
- 用户会把 `gogo-app` 理解成 `Pi` 的壳，而不是知识库产品
- 许多命令与知识库场景无关，只会增加噪音
- 命令语义受 `Pi` 演进影响更大，产品稳定性更弱
- 前端需要处理更多 runtime-specific 反馈、状态和失败分支
- 文档、帮助、自动补全、权限边界都会更复杂

结论：

不适合作为 `gogo-app` 的默认路线。

### B. 只支持 knowledge-base skills

优点：

- 产品边界最清晰：slash 命令就是知识库能力入口
- 更符合 `gogo-app` 作为知识库工作台的定位
- 用户会把注意力放在“如何增强知识库能力”，而不是“如何操作底层 Agent”
- 命令命名可以围绕业务场景设计，更好理解
- 更适合做自动补全、命令说明、示例和权限控制
- 更容易沉淀成团队自己的工作流资产

缺点：

- 无法覆盖 `Pi` 的全部原生命令
- 某些系统级操作仍然需要通过别的 UI 或终端完成
- 需要明确 skill 的注册、发现、参数输入和错误反馈机制

结论：

这是当前最推荐的默认方案。

### C. 同时支持 Pi 命令 + skills

优点：

- 理论上能力最全
- 高级用户可以同时获得 runtime 控制和知识库工作流

缺点：

- 两套命令体系会混在一起，用户难以建立稳定心智
- 自动补全列表会变长，难以排序和解释
- 文档、帮助和实现复杂度最高
- 很容易把简单输入框变成“半个 IDE 命令面板”

结论：

不适合第一阶段。  
这条路线只有在产品已经非常成熟、并且确实需要高级双层命令空间时才值得考虑。

### D. 只支持 schemas

优点：

- 结构化、标准化，便于沉淀知识资产
- 对“录入型命令”很有帮助

缺点：

- schema 更像输入约束，不等于完整技能
- 如果只有 schema，没有 skill，用户会觉得“命令只是打开一个表单”
- 对聊天体验的即时价值通常不如 skill 明显

结论：

更适合作为 skill 的补充机制，而不是 slash 命令的唯一来源。

## 推荐决策

当前推荐：

- `gogo-app` 默认只暴露 knowledge-base 中的 `skills`
- 不直接暴露 `Pi agent` 内置 slash 命令
- `schemas` 作为 skill 的参数结构或后续增强层存在，不单独作为第一批 slash 命令主来源

一句话总结：

> `gogo-app` 的 slash 命令应该代表“知识库能力”，而不是“Pi runtime 控制面”。

## 为什么这个决策更适合 gogo-app

### 1. 保持产品聚焦

`gogo-app` 的核心价值不是让用户更方便地操作 `Pi`，而是让用户更方便地使用、组织和增强知识库。

如果把 `Pi` 内置命令整体暴露出来，用户会越来越关注：

- 模型怎么切
- runtime 怎么控
- agent 命令怎么用

而不是关注：

- 知识怎么写回
- 结构化技能怎么积累
- 团队工作流怎么沉淀

### 2. 更利于知识库本身演进

如果 slash 命令只来自 skills，那么每新增一个命令，本质上都是在增强 knowledge-base：

- 新增一个技能
- 为技能写说明
- 为技能定义输入约束
- 为技能设计输出模板

这会让命令体系成为知识库资产的一部分，而不是外来系统能力的镜像。

### 3. 降低不必要复杂度

Pi 内置命令往往带有明显的系统性、运行时性和环境依赖。

这些能力不是不能做，而是不应该默认进入 `gogo-app` 的主交互面。  
否则会带来：

- 更复杂的自动补全
- 更多异常状态
- 更多“桌面版能不能做 / Web 版能不能做”的差异
- 更多需要解释的底层概念

### 4. 便于形成统一 UI 心智

如果 slash 命令只表示 skills，那么用户很容易理解：

- 输入 `/` 看到的是“这套知识库会做什么”
- 不是“底层 agent 支持什么命令”

这种心智更稳定，也更容易扩展到：

- 命令说明
- 参数面板
- 示例输入
- 团队共享技能目录

## 对 schemas 的位置建议

推荐把 schemas 放在 skill 之后，而不是与 skill 并列为第一层 slash 来源。

更合适的关系是：

- skill 决定“做什么”
- schema 决定“这个 skill 的结构化输入长什么样”

例如：

- `/write-wiki-note`
  - 背后可以引用 `wiki_note.schema.json`
- `/paper-card`
  - 背后可以引用 `paper_card.schema.json`

这样用户看到的是一个清晰的命令，而不是直接面对 schema 名称本身。

## 可选折中方案

如果后续真的有少数 `Pi` 命令必须在 `gogo-app` 中出现，建议不要混进主 slash 列表，而是单独处理：

- 通过设置面板、工具按钮或独立入口暴露
- 或者放在“高级 / 实验性命令”分组里，默认隐藏

换句话说：

- 主 slash 列表：只放 knowledge-base skills
- 高级 runtime 能力：走别的入口

这样可以避免命令空间污染。

## 第一阶段建议

第一阶段建议只做下面这些：

1. `/` 只拉起 skills 自动补全
2. 每个 skill 显示：
   - 命令名
   - 一句话说明
   - 是否需要结构化参数
3. 选择后：
   - 直接插入命令草稿
   - 或弹出最小参数表单
4. 暂不接入 Pi 内置命令
5. 先接入 knowledge-base 的 skills 与 schemas，但 schema 参数面板后置

## 当前实现状态

当前产品实现已经落地了第一版最小方案：

- 聊天输入框输入 `/` 时，会拉起知识库 slash 自动补全
- 同时提供一个额外按钮打开同一套命令列表
- 列表同时包含当前 knowledge-base 的 `skills` 与 `schemas`
- 面板会按 `Skills / Schemas` 分组，并用类型徽标做区分
- 选择某个命令后，默认把 `/<command-name>` 插入输入框草稿
- 不会直接发送

当前版本还没有做的部分：

- schema 驱动的参数面板
- skills / schemas 的结构化参数编辑
- Pi 内置 slash 命令接入

也就是说，当前实现是：

> slash 命令已经作为 knowledge-base 能力的轻量入口落地，但仍然保持“插入草稿、由用户确认发送”的保守策略。

## Slash 命令在 UI 上应该怎么呈现

如果采用“slash 命令只来自 knowledge-base skills”这条路线，那么 UI 也应该围绕这个边界来设计。

核心原则是：

- slash 面板应当像“知识库能力选择器”，而不是“系统命令终端”
- 默认帮助用户发现 skill，而不是帮助用户记忆命令语法
- 能插入草稿就不要直接执行，避免误触

### 推荐的第一层交互

推荐交互：

1. 用户在聊天输入框输入 `/`
2. 弹出一个轻量自动补全面板
3. 面板只显示当前知识库已注册的 skills
4. 选择后将命令插入输入框，默认不自动发送

每个候选项建议显示：

- slash 命令名
- 一句话说明
- 可选标签：`需要参数`、`写回`、`分析`、`结构化输出`

不建议第一阶段显示：

- 太多技术字段
- 底层 runtime 信息
- `Pi` 内部命令来源

### 为什么推荐“插入草稿”而不是“直接执行”

优点：

- 更安全，不容易误触
- 用户仍然保留对最终 prompt 的控制权
- 更符合当前聊天框的工作方式
- 更容易与已有的引用、写回、ingest 等提示词机制统一

缺点：

- 多一步确认
- 对极高频命令来说略慢

结论：

第一阶段推荐“插入草稿，不直接执行”。  
以后如果某些 skill 非常稳定，再单独评估是否支持“一键执行”。

### 对需要参数的 skill 怎么处理

建议分两类：

1. 无参数或参数极少的 skill
   - 直接插入命令草稿
   - 例如：`/write-wiki-note`

2. 明显依赖结构化输入的 skill
   - 点击后先弹最小参数面板
   - 完成后再生成草稿

参数面板应尽量轻：

- 不要做成完整设置页
- 只显示 schema 中最关键的 1 到 3 个字段
- 保持“填完就回到聊天输入框”这条主路径

### 自动补全面板建议

建议面板具备：

- 键盘上下选择
- `Enter` 选中
- `Esc` 关闭
- 支持按命令名和说明模糊搜索

建议面板不要一开始就做：

- 多级分组树
- 复杂筛选器
- 图标墙式入口

第一阶段更适合一个简单、稳定、快速的列表。

### 与聊天输入框的关系

slash 命令不应该变成一个脱离聊天的独立系统。  
更合适的关系是：

- slash 只是聊天输入的一种快捷构造方式
- 最终仍然回到同一个聊天草稿框
- 与引用、写回、ingest 提示词共用同一条输入链路

这样用户会自然理解：

- 引用是在往草稿里插入上下文
- 写回是在往草稿里插入意图
- slash 是在往草稿里插入一个 skill 模板

### 推荐的视觉风格

推荐：

- 浮层窄而轻
- 贴近输入框出现
- 以文本信息为主
- 视觉重点放在命令名和一句话说明

不推荐：

- 做成全屏命令面板
- 做成复杂 IDE palette
- 做成设置页式的大表单

因为这会让 slash 从“聊天增强”变成“另一套操作系统”。

### 与 schema 的关系

如果某个 skill 背后有 schema，UI 不应先暴露 schema 名，而应表现为：

- 用户先选 skill
- 只有当这个 skill 需要结构化参数时
- UI 才按 schema 生成最小参数输入区

也就是说：

- 用户看见的是“skill”
- schema 只作为其背后的结构层

### 最推荐的第一版 UI

第一版最推荐做法：

1. 聊天输入框输入 `/`
2. 弹出 skills 自动补全列表
3. 选中后把命令插入草稿
4. 如果 skill 需要参数，先弹最小参数面板再插入草稿
5. 用户自己决定是否发送

一句话总结：

> slash UI 应该像“知识库技能选择器”，而不是“Pi 命令控制台”。

## 最终建议

最终建议是：

- `gogo-app` 不要试图成为 `Pi` 的通用 slash 命令前端
- `gogo-app` 应该把 slash 命令定义为 knowledge-base 的技能入口
- `Pi` 继续作为底层 agent/runtime
- knowledge-base 的 `skills` 才是用户在产品层感知到的命令系统

这条路线的最大好处不是“实现更简单”，而是：

**它更符合产品定位，也更容易让用户把注意力放在知识库本身。**
