# gogo-app 桌面应用封装方案评估

**最后更新**: 2026-04-16

> 本文档评估 `gogo-app` 封装为桌面应用的可选路线，重点关注：
> - 本地知识库目录访问
> - Pi CLI / RPC 集成
> - 设置面板中的 Provider / 登录桥
> - 后续 Wiki Markdown 编辑、slash 命令、文件监听等本地能力
>
> 当前仓库已经决定把桌面化主路线定为 Tauri，实施方案见 [tauri-migration-plan.md](tauri-migration-plan.md)。
> 本文档保留的 Electron / Wails 对比主要用于方案比较，不再代表当前仓库推荐继续走这些路线。

## 1. 背景

`gogo-app` 当前是本地 FastAPI + 浏览器前端的形态：

```text
Desktop User
  -> Browser
  -> gogo-app frontend
  -> FastAPI
  -> Pi RPC / knowledge-base local files
```

这套结构已经证明产品链路可行，但在以下场景里明显受到“纯 Web”形态限制：

- 不能稳定选择本地目录绝对路径
- 不能自然拉起 Pi CLI 做 `/login`
- 不适合做本地 Markdown 编辑、文件监听、外部修改检测
- slash 命令若要桥接 Pi 原生命令，纯 Web 会比较别扭

因此桌面化的目标不是“把网页包起来”这么简单，而是：

- 保留当前前后端主体
- 增加桌面侧原生能力桥
- 不让现有架构推倒重来

## 2. 评估标准

这里主要按这几个维度比较：

- 开发成本：现有代码迁移难度、团队上手成本
- 与当前架构兼容性：是否容易复用 FastAPI + 前端 + Pi RPC
- 本地能力：目录选择、文件监听、进程管理、调起 Pi CLI
- 体积与性能：安装包大小、内存、启动速度
- 跨平台：macOS / Windows / Linux 可交付性
- 长期维护：后续功能演进和调试成本

## 3. 可选方案

### 3.1 Electron

#### 方案描述

使用 Electron 作为桌面壳，前端继续使用现有 HTML/CSS/JS，桌面主进程负责：

- 启动和托管本地 FastAPI
- 打开应用窗口
- 调用系统文件夹选择器
- 提供与 Pi CLI、文件系统、日志目录相关的原生桥接

#### Pros

- 对当前前端最友好：可以几乎原样复用现有页面
- JavaScript / TypeScript 生态成熟，资料最多
- 原生能力集成直接：目录选择、打开文件、调起子进程都很成熟
- 非常适合后续加：
  - Pi CLI 登录桥
  - Wiki 本地编辑
  - 文件拖拽、监听、自动刷新
  - slash 命令桥接
- 对“前端为主、后端独立进程”这种结构包容性很好

#### Cons

- 安装包和运行时较重
- 内存占用通常高于 Tauri / Wails
- 需要额外注意 Electron 安全边界
- 如果未来只做一个相对轻量的本地知识库工作台，Electron 可能显得偏重

#### 对 gogo-app 的匹配度

很高，但已不再是当前仓库采用的路线。  
如果只做历史比较，Electron 仍然是“最快封装现有 Web 前端”的典型方案；但对当前仓库来说，这条路已经被 Tauri 取代。

### 3.2 Tauri

#### 方案描述

使用 Tauri 作为桌面壳，前端仍然复用现有页面，原生侧由 Rust 提供：

- 应用窗口
- 系统文件选择
- 进程桥接
- 原生命令调用

常见实现方式是：

- 前端继续访问本地 FastAPI
- 或者把部分桌面桥逻辑放进 Tauri command

#### Pros

- 安装包更小、资源占用更低
- 原生感更强，长期分发体验更好
- 对本地文件系统、命令调用、系统能力的支持也足够强
- 很适合做“桌面工作台 + 本地文件编辑 + 本地 agent 调度”这一类应用

#### Cons

- 引入 Rust 工具链，团队学习成本更高
- 与 Electron 相比，工程调试和资料生态稍弱
- 对“现有 Web + Python 服务”的整合需要更多工程设计
- 如果团队主要熟悉 Python / JS，前期会慢一些

#### 对 gogo-app 的匹配度

高。  
如果你更重视长期产品化、轻量分发和本地工作台体验，Tauri 很有吸引力。  
但它不是“最快封装”的路线，而是“更适合长期桌面产品”的路线。

### 3.3 Wails

#### 方案描述

使用 Wails 作为桌面壳，前端依旧是 Web UI，原生层由 Go 提供。  
它适合希望用 Go 写桌面桥，而不想引入 Electron 或 Rust 的团队。

#### Pros

- 比 Electron 更轻
- 对“前端 + 本地原生桥”的模式比较友好
- Go 侧做进程管理、文件系统操作、系统桥接会比较顺手

#### Cons

- 相比 Electron / Tauri，生态和社区更小
- 现有 `gogo-app` 主后端是 Python，不是 Go，无法天然复用语言栈
- 如果不是明确想把桌面桥放在 Go 里，它不会带来特别明显的优势

#### 对 gogo-app 的匹配度

中等。  
不是不行，但也不是当前最自然的路线。  
除非后续希望桌面壳和本地桥更多走 Go，否则优先级通常低于 Electron / Tauri。

### 3.4 保持浏览器前端，新增本地桌面 Launcher / Helper

#### 方案描述

不立刻做完整桌面壳，而是增加一个本地启动器或 helper，负责：

- 启动 FastAPI
- 打开浏览器
- 提供少量本地桥 API
- 未来再逐步升级成完整桌面应用

#### Pros

- 迁移成本最低
- 可以最快验证“桌面能力到底需要哪些”
- 对当前代码改动最小
- 适合作为 Electron / Tauri 前的过渡阶段

#### Cons

- 产品形态不完整，看起来仍像“本地服务 + 浏览器页”
- 原生窗口、菜单、托盘、统一设置体验都比较弱
- Pi CLI 登录桥、Markdown 编辑、目录选择等能力最终还是要重新设计成更正式的桌面交互

#### 对 gogo-app 的匹配度

中等偏高，适合作为过渡方案。  
如果当前最重要的是尽快验证桌面能力而不是正式分发，这条路很实用。

### 3.5 直接重写为原生桌面 UI（PySide / Qt 等）

#### 方案描述

放弃当前 Web UI，直接用原生桌面 UI 框架重写工作台。

#### Pros

- 文件系统、本地编辑、进程桥接都很自然
- 理论上可获得更强的桌面原生感

#### Cons

- 基本等于重做前端
- 现有 HTML/CSS/JS 资产复用率低
- 开发和维护成本最高
- 不符合当前 `gogo-app` 以 Web 工作台为中心的产品形态

#### 对 gogo-app 的匹配度

低。  
除非未来产品方向彻底改变，否则不建议走这条路。

## 4. 汇总对比

| 方案 | Pros | Cons | 对 gogo-app 的建议 |
|---|---|---|---|
| Electron | 最容易复用现有前端；原生能力成熟；最快做原型 | 较重、安装链依赖额外二进制下载 | 仅保留作历史对比，不再是当前路线 |
| Tauri | 轻量、安装链更稳、长期产品化更优 | 引入 Rust，前期成本更高 | 当前已采用的桌面化主路线 |
| Wails | 比 Electron 轻，Go 侧本地桥顺手 | 生态较小，和现有 Python 主后端不天然贴合 | 可选，但优先级低于 Electron / Tauri |
| 浏览器 + Helper | 改动最小，适合过渡验证 | 产品形态不完整，很多桌面问题只是延后 | 适合作为过渡阶段 |
| 原生重写 | 原生桌面能力强 | 成本最高，几乎重做 | 不建议 |

## 5. 推荐路线

### 5.1 如果目标是尽快落地

推荐：

1. 直接沿当前仓库已经落地的 Tauri 路线继续推进
2. 尽快验证这三件事：
   - 原生知识库目录选择
   - Pi CLI `/login` 登录桥
   - Wiki Markdown 编辑 / 文件监听

这条路线的价值在于：

- 现有产品几乎不用大改
- Tauri 第一阶段已经跑通，不需要再回退到别的桌面壳
- 能继续围绕已经存在的桌面 bridge 和本地后端托管能力迭代

### 5.2 如果目标是正式产品化

推荐：

1. 继续用 Tauri 完成 Pi CLI 登录桥、Markdown 编辑和打包分发
2. 把 Electron 保留为历史对比，不再作为主实现分支

原因是：

- `gogo-app` 当前已经是“本地 Web 工作台 + 本地服务 + 本地 CLI”结构
- Tauri 已经证明对这类架构同样适配，而且当前仓库就是按这条线实现的
- 桌面版的关键能力不是 UI 重写，而是桌面桥接

### 5.3 如果愿意接受更高前期工程成本，换取更好的长期形态

可以直接选 Tauri。  
但前提是团队接受：

- Rust 工具链
- 更复杂一些的工程整合
- 初期开发速度略慢

## 6. 建议的分阶段实施

### Phase 1: 桌面能力最小闭环

- 封装应用窗口
- 启动本地 FastAPI
- 提供原生目录选择
- 提供“打开日志/状态目录”能力

### Phase 2: Pi CLI 深度集成

- 实现设置面板里的 `desktop-login` 接口
- 拉起交互式 `pi`
- 引导用户执行 `/login`
- 登录完成后刷新 provider / model 状态

### Phase 3: 本地编辑工作台

- Wiki Markdown 编辑模式
- 外部修改检测
- 自动保存或保存草稿
- 与知识库写回规范逐步对齐

### Phase 4: 命令与高级工作流

- 输入框 slash 命令按钮
- 部分命令桥接 Pi 原生命令
- 更完整的 diagnostics / logs / session tools

## 7. 最终建议

如果只给一个当前建议：

- **当前主路线：Tauri**
- **Electron：只保留作历史方案比较**

对当前 `gogo-app` 来说，最重要的不是选一个“最时髦”的桌面框架，而是选一个能最快把下面这些能力稳定交付出来的方案：

- 选择本地知识库目录
- 调用 Pi CLI `/login`
- 支持本地 Markdown 编辑
- 支持文件监听和更强的工作台能力

从当前仓库状态看，Tauri 已经是第一步，而且已经跑通了最小闭环；Electron 不再作为后续实施方向。
