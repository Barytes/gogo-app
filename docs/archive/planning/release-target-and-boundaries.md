# gogo-app 发布目标与边界

**最后更新**: 2026-04-17

> 本文档用于明确 `gogo-app` 当前对外发布时的边界：  
> 目标用户是谁；  
> 当前已经能承诺什么；  
> 还不能承诺什么；  
> 从当前开发态走到“普通用户可直接安装”的正式桌面版还差哪些条件。

## 0. 当前对外目标

`gogo-app` 当前已经明确把对外阶段目标收敛为：

- **面向普通最终用户**
- **以 Windows / macOS 可直接安装的桌面版为主形态**
- **不把“自行搭开发环境”作为正式交付前提**

这意味着：

- Web 版和桌面开发版仍然重要，但它们现在主要承担开发、验证和过渡职责
- “最终用户双击可安装的桌面版”不是远期愿景，而是当前对外发布的目标标准
- README、安装文档、打包任务、首次启动引导、companion knowledge-base 都应围绕这个目标收敛

## 0.1 已明确的首发决策

截至 **2026-04-17**，当前已经明确的首发决策包括：

- 首发平台必须**同时支持 Windows + macOS**
- 首发正式产品只认**桌面版**；Web 版不作为正式对外产品形态
- 用户安装完成后的主路径是：
  1. 检测并安装 `pi`
  2. 配置模型/API key（允许跳过）
  3. 进入 companion knowledge-base
  4. 若已完成模型配置，则可以继续上传 / ingest / 聊天 / 写回；若未配置，则至少可以浏览 Wiki
- companion knowledge-base 会随安装包提供，并在安装/首次启动过程中让用户决定其安装路径
- 用户后续可以切换到自己的 knowledge-base，同时 companion knowledge-base 仍保留为可随时切回的示例库
- 首发模型配置支持：
  - API key 型 provider
  - `pi` 已稳定支持且桌面引导已验证通过的 OAuth provider
- API key 首发策略是**仅保存在本机认证文件中，不自动上传，暂不接入 macOS Keychain / Windows Credential Manager**
- 首发最低成功标准是：用户完成模型配置后，能够跑通一次**上传 -> ingest -> 聊天 -> 写回**
- 首发必须支持**中文路径**与**带空格路径**
- 知识库浏览与编辑可以断网；聊天链路依赖用户自己配置并联网访问的模型
- 默认日志/诊断策略是：**只保存在本地，不自动上传；用户可手动导出后发 GitHub issue**
- 当前正式支持渠道为 **GitHub Issues**

## 1. 结论先行

截至 **2026-04-17**，`gogo-app` 当前的**发布目标**与**实际状态**需要分开描述。

发布目标是：

- 一个**面向普通最终用户、可直接安装使用的本地 knowledge-base 桌面应用**

当前实际状态是：

- 一个**可从源码运行的本地 knowledge-base 工作台**
- 一个**可供开发者与小范围内测用户运行和验证的 Tauri 桌面开发版**
- 一个**已经打通 Tauri 发布态资源映射、独立后端 runtime 打包，并可在 macOS 上产出 `.app/.dmg` 基础 bundle 的桌面发布原型**
- **还不是**一个可以面向普通最终用户稳定分发的“安装包桌面应用”

换句话说，当前仓库可以承诺的是：

- 开发者可以按文档准备本地环境，从源码启动 Web 版
- 开发者可以按文档准备本地环境，从源码启动 Tauri 桌面开发版
- 产品方向已经明确收敛到“普通用户可直接安装”的桌面版交付目标

当前仓库**不能承诺**的是：

- 用户在未安装开发环境的情况下，下载一个 Windows / macOS 安装包后即可直接使用
- Windows / macOS 两侧都已经完成最终用户级别的独立后端 runtime 验收
- 安装包已经完成 Windows 代码签名或 macOS notarization
- 安装包在干净机器上完成过面向最终用户的稳定回归

## 2. 两种发布层级

### 2.1 开发者可运行的桌面版

这里指的是：

- 从仓库源码启动
- 本机自行安装 Node、Rust、Python 等开发依赖
- 本机自行准备 knowledge-base 目录
- 当前桌面版会优先使用随包分发的 bundled `pi` 运行目录；若既没有 bundled `pi`，又没检测到系统 `pi`，才会在启动界面引导 fallback 安装
- 通过 `npm run desktop:dev` 或等价开发命令运行

它的定位是：

- 内部开发
- 中间验收版本
- 安装链路落地前的能力验证
- 产品与架构迭代

它**不是**“下载后双击安装即可用”的最终交付物。

### 2.2 最终用户双击可安装的桌面版

这里指的是：

- 用户无需克隆源码仓库
- 用户无需安装 Node、Rust、或手动运行 `uvicorn`
- 用户通过标准安装介质完成安装
  - Windows：首发先提供 `NSIS setup.exe`
  - macOS：如 `.app + dmg`
- 应用启动后能够以面向用户的方式完成：
  - 静默检测并安装 `pi`
  - 首次启动引导
  - 模型/API key 配置（可跳过）
  - companion knowledge-base 进入与示例体验
  - knowledge-base 切换
  - 后端运行时托管
  - 基本诊断与故障提示

这才是对外“正式桌面版”的标准。  
同时，这也是 `gogo-app` 当前对外阶段的目标形态。

## 3. 当前支持范围

| 形态 | 当前状态 | 适用对象 | 当前要求 |
|---|---|---|---|
| Web 版（源码启动） | 已支持，但不是正式对外产品形态 | 开发者 / 技术用户 | Python 环境、`uv` 或可用 `python + uvicorn`、本地 knowledge-base |
| Tauri 桌面开发版（源码启动） | 已支持，但属于开发态 | 开发者 / 内测用户 | Node `22/24`、Rust、Python 环境、本地 knowledge-base；若未提供 bundled / system `pi`，fallback 安装链路仍需要 `npm` |
| Windows 最终用户安装包 | 目标明确，但未完成 | 普通最终用户 | 首发安装介质策略已收敛为 `NSIS setup.exe`，但当前仍不应承诺可直接使用 |
| macOS 最终用户安装包 | 目标明确，但未完成 | 普通最终用户 | 当前仍不应承诺可直接使用 |

当前建议的对外口径是：

- **发布目标**：面向普通用户提供可直接安装的 Windows / macOS 桌面版
- **当前已支持**：从源码运行的 Web 版和桌面开发版
- **当前正式产品方向**：桌面版；Web 版不作为正式对外交付
- **当前正在建设**：Windows / macOS 的最终用户安装包链路
- **当前仍未达到**：面向普通用户的“一键安装即用”交付标准

## 4. 当前外部依赖

### 4.1 所有形态共用的运行前提

无论是 Web 版还是桌面开发版，当前都依赖这些外部条件：

- 一个可读取的 knowledge-base 目录
  - 至少需要 `wiki/`
  - 至少需要 `raw/`
- Pi CLI 运行时
  - 聊天主链路依赖 `pi --mode rpc`
  - OAuth 主路径依赖本机 `pi` 与 `/login`
  - 当前桌面版已支持随包分发 bundled `pi` 运行目录，并会优先使用它；若缺失 bundled/system `pi`，才会在启动界面把 `pi` 安装到 app data 下的托管目录
  - 在 macOS 上，bundled `pi` 不能作为一个裸下载目录直接分发；应作为 `gogo-app.app` 内嵌运行时一起签名，并纳入整包 notarization
- 对应模型所需的认证状态
  - API key
  - 或 Pi 托管的 OAuth 登录状态

### 4.2 开发者可运行桌面版的额外依赖

当前桌面开发版还要求本机自行准备：

- Node `22` 或 `24`
- `npm install`
- Rust 工具链
- Python `>=3.9`
- `uv`，或可用的 `python3/python + uvicorn`
- 平台原生桌面依赖

当前桌面后端启动策略仍然会在本机寻找：

1. `GOGO_DESKTOP_PYTHON`
2. 仓库内 `.venv/bin/python`
3. `uv run uvicorn ...`
4. `python3 -m uvicorn ...`
5. `python -m uvicorn ...`

这说明当前桌面版仍然是**开发态运行模型**，而不是自包含发行包。

### 4.3 最终用户安装包应当达到的依赖边界

当 `gogo-app` 进入“最终用户双击可安装桌面版”阶段时，目标边界应当是：

- 不再依赖源码仓库目录
- 不再要求用户自行安装 Node / Rust
- 不再要求用户手动进入仓库执行 `uv run uvicorn`
- 不再依赖开发态 `.venv`
- 安装包应当自己托管应用资源与后端运行时

以下依赖是否继续保留，需要在正式发版前明确：

- companion knowledge-base 的安装路径由谁决定
- `pi` 的静默安装与升级链路如何随安装器交付
- API key 是否以及何时升级到 macOS Keychain / Windows Credential Manager

## 5. 当前已知限制

以下限制在对外文档里应该明确写出，而不是隐含假设：

1. 当前仓库的 Tauri 桌面能力仍以**开发态运行**为主，不等于已经完成正式分发。
2. 当前 macOS 调试构建已包含独立后端 runtime，但 Windows 侧构建、跨平台验收与最终安装器整合仍未完成。
3. 当前虽然已能产出 macOS `.app/.dmg` 基础 bundle，但尚未完成 **Windows / macOS 最终用户安装包** 的验收闭环。
4. 当前尚未完成**Windows 代码签名**与 **macOS notarization**；这不仅影响主应用，也影响 bundled `pi` 这类内嵌可执行文件的最终分发可用性。
5. 当前尚未完成**干净机器验证**，因此不能把“开发机可运行”等同于“最终用户可使用”。
6. 当前已支持将 companion knowledge-base 随桌面 bundle 资源交付，并在发布态默认 provision 到可写目录；当前跨平台实现是“首次启动时由用户选择路径并记住结果”，而不是在不同平台安装器 UI 中分别实现一套定制路径选择。
7. 当前聊天与 OAuth 登录链路仍依赖本机 `pi`，这属于真实外部依赖，不应在 README 中省略。
8. 首发虽然计划支持 API key 与部分 OAuth，但“`pi` 已稳定支持且桌面引导已验证通过”的 provider 范围尚未列成正式兼容清单。
9. 当前自动更新尚未实现。
10. 当前尚未完成“中文路径 / 空格路径”在正式安装包链路中的跨平台验收。
11. 调试构建与开发态不能混为一谈：打包后的 debug `.app` 也必须走 bundle 内后端，而不是误连本地 dev server；这一点已经在当前 Tauri 实现里修正。
12. 首发阶段的 API key 仅保存在本机认证文件中；当前尚未接入 macOS Keychain 或 Windows Credential Manager。

## 6. 对外文案建议

为了避免误导，当前 README、介绍页或对外交流中建议使用以下表述：

- `gogo-app` 的对外目标是面向普通用户提供可直接安装的桌面版。
- 当前仓库已提供 Web 版源码运行体验，以及 Tauri 桌面开发版，用于产品迭代和安装链路落地前的验证；正式产品方向仍然是桌面版。
- Windows / macOS 的最终用户安装包仍在建设中，当前版本还不能宣称已经达到普通用户可直接安装即用的标准。
- 若要完整体验聊天和 OAuth 登录链路，当前仍需要本机可用的 `pi`；正式安装包目标是随安装器检测并静默安装。
- 首发桌面版会随应用提供 companion knowledge-base，并允许用户切换到自己的 knowledge-base。
- 问题反馈与支持统一通过 GitHub Issues 进行。

当前**不建议**使用的表述包括：

- “已经提供桌面安装包，可直接发给普通用户使用”
- “无需任何外部依赖即可运行”
- “桌面版已经产品化完成”

## 7. 从开发者桌面版升级到最终用户安装包的验收条件

当且仅当以下条件都满足后，才适合把 `gogo-app` 对外表述升级为“最终用户可安装桌面版”：

1. Tauri bundle 已启用，并能稳定产出 Windows / macOS 安装介质。
2. 应用资源与后端运行时已随安装包交付，不再依赖源码目录。
3. knowledge-base 接入流程已明确：
   - companion knowledge-base 随安装包提供
   - 安装时由用户决定其路径
   - 用户可切换到自己的 knowledge-base，并可随时切回 companion knowledge-base
4. `pi` 依赖的静默安装、可见性、登录与失败提示已形成可操作引导；当前仓库只完成了“桌面版应用内检测 + 托管安装”的过渡形态。
5. API key 已明确为“仅保存在本机认证文件中，不自动上传”的首发策略，且文档已清楚标注当前未接入系统级钥匙串。
6. 应用内具备最小可用的 diagnostics 与日志导出能力，且默认只保存在本地。
7. 首发支持矩阵已经明确：
   - Windows + macOS 同时可用
   - 中文路径与空格路径通过验收
   - API key provider 与受支持的 OAuth provider 通过验收
8. 在至少一台干净 Windows 和一台干净 macOS 上验证通过。
9. 完成对应平台的签名与分发安全要求：
   - Windows 代码签名
   - macOS Developer ID + notarization

在此之前，`gogo-app` 当前最准确的状态仍然是：

**一个以“普通用户可直接安装桌面版”为对外目标、但当前仍主要通过源码运行与桌面开发版推进落地的本地 knowledge-base 工作台。**
